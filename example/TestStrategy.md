# TestStrategy

本流程定义了测试策略制定的核心步骤。当有需求PRD来了以后，AI应该按照以下流程制定测试策略。

---

## 输入输出

**输入**：
- PRD 文档
- 技术方案文档
- 项目背景和目标

**输出**：
- 测试策略文档（命名格式：`ModuleName_ID_Strategy.md`）

---

## 测试策略制定流程

### 第一步：分析质量目标

**输入**：
- 来自 PRD 的"非功能性需求"章节（性能、可靠性、安全性、易用性、效率、维护性、可移植性）
- 产品的质量期望（来自产品经理或业务方）

**任务清单**：
- [ ] 从 PRD 中提取所有质量目标
- [ ] 将质量目标按优先级分类（关键、重要、一般）
- [ ] 明确每个质量目标的可量化指标（如：响应时间 < 100ms、99.9% 可用性、并发用户数 ≥ 1000）
- [ ] 识别质量目标之间的潜在冲突（如：性能 vs 安全性），并确定优先级

**输出**：质量目标清单（包含优先级和量化指标）

---

### 第二步：风险分析

**风险类型**：
1. **功能风险**
   - 需求是否清晰？
   - 是否有需求变更历史？
   - 是否有复杂业务逻辑？
   - 是否有性能要求？

2. **技术风险**
   - 是否使用了新技术？
   - 是否有技术选型争议？
   - 是否有依赖的第三方服务？
   - 是否有接口兼容性问题？

3. **项目风险**
   - 时间是否紧迫？
   - 资源是否充足？
   - 团队成员是否稳定？
   - 是否有依赖的外部因素？

**任务清单**：
- [ ] 进行全面的风险分析
- [ ] 识别所有可能的风险项
- [ ] 评估每个风险的发生概率和影响程度（1-5分）
- [ ] 计算风险值 = 发生概率 × 影响程度
- [ ] 按风险值从高到低排序风险项
- [ ] 针对高风险项制定风险应对策略（规避、转移、减轻、接受）

**输出**：风险分析报告（包含风险列表、风险值排序、应对策略）

---

### 第三步：确定测试范围和重点

**任务清单**：
- [ ] 基于 PRD 的"用户故事与范围"章节，明确需要测试的功能
- [ ] 基于 PRD 的"范围外"章节，明确不需要测试的功能
- [ ] 基于"风险分析报告"，确定测试重点：
  - 高风险功能 → 重点测试
  - 关键质量目标 → 重点测试
  - 核心业务流程 → 重点测试

**测试范围定义示例**：
```markdown
## 测试范围

### 范围内
- 用户登录功能
- 用户注册功能
- 密码重置功能
- 用户信息管理功能

### 范围外
- 第三方登录功能（暂未实现）
- 用户头像上传功能（后续版本）

## 测试重点
- 用户登录安全性（高风险）
- 密码重置流程（关键业务）
- 用户信息一致性（质量目标）
```

**输出**：测试范围和重点清单

---

### 第四步：选择测试类型

**测试类型选择矩阵**：

| 测试类型 | 适用场景 | 覆盖率目标 | 负责人 | 工具 |
|---------|---------|-----------|---------|------|
| 单元测试 | 代码级逻辑验证 | ≥ 80% | 开发工程师 | Jest、PyTest、JUnit |
| 集成测试 | 模块间接口验证 | ≥ 70% | 开发工程师 | Supertest、Postman |
| 系统测试 | 完整功能流程验证 | 关键路径100% | 测试工程师 | Playwright、Cypress、Selenium |
| 性能测试 | 性能指标验证 | 目标指标100% | 测试工程师 | JMeter、K6、Locust |
| 安全测试 | 安全漏洞检测 | 关键漏洞0 | 安全工程师 | OWASP ZAP、SonarQube |
| AI模型测试 | AI系统专项测试 | 根据项目特点自定义（常见指标：数据覆盖率≥80%、模型鲁棒性测试通过率≥90%、公平性指标Demographic Parity Difference≤0.1、异常输入防御率≥95%） | AI测试工程师 | Evidently、Fairlearn、SHAP |
| 探索式测试 | 发现未知缺陷 | 时间驱动 | 测试工程师 | 手工测试 |

**任务清单**：
- [ ] 根据质量目标和风险分析，选择合适的测试类型
- [ ] 确定每种测试类型的负责人
- [ ] 确定每种测试类型使用的工具
- [ ] 定义每种测试类型的覆盖率目标
- [ ] 对于AI项目，额外添加AI模型测试类型

**输出**：测试类型清单（包含覆盖率目标）

---

### 第五步：确定测试深度和广度

**测试深度**：指测试的深入程度（如：只测试正常流程，还是测试异常流程、边界条件）

**测试广度**：指测试的覆盖范围（如：测试多少功能、多少场景）

**任务清单**：
- [ ] 基于"测试重点"，确定每个重点功能的测试深度：
  - 关键功能 → 深度测试（正常流程 + 异常流程 + 边界条件 + 压力测试）
  - 重要功能 → 中等深度测试（正常流程 + 异常流程）
  - 一般功能 → 浅度测试（正常流程）
- [ ] 基于"测试范围"，确定测试广度：
  - 高风险功能 → 广度覆盖（所有使用场景）
  - 核心业务流程 → 广度覆盖
  - 边缘功能 → 精简覆盖

**测试深度和广度示例**：
```markdown
## 测试深度和广度

### 用户登录功能（重点功能）
- 测试深度：深度测试
  - 正常流程：正确的用户名和密码
  - 异常流程：错误的用户名、错误的密码、账号锁定、验证码过期
  - 边界条件：用户名长度边界、密码长度边界、特殊字符
  - 压力测试：并发登录 1000 用户
- 测试广度：广度覆盖
  - Web 登录、移动端登录、API 登录
  - 不同浏览器（Chrome、Firefox、Safari）
  - 不同设备（PC、平板、手机）

### 用户信息管理功能（一般功能）
- 测试深度：浅度测试
  - 正常流程：查看信息、修改信息
- 测试广度：精简覆盖
  - 仅 Web 端，仅 Chrome 浏览器
```

**输出**：测试深度和广度定义

---

### 第六步：安排测试活动

**测试活动安排原则**：
1. **测试左移**：尽可能早地进行测试（如：单元测试、集成测试）
2. **分层测试**：按测试金字塔进行（单元测试 > 集成测试 > 系统测试）
3. **风险优先**：高风险功能优先测试
4. **迭代推进**：与开发迭代同步进行

**测试活动时间表示例**：

| 阶段 | 测试活动 | 负责人 | 工具 | 时间 |
|------|---------|---------|------|------|
| 测试阶段 | 系统测试 | 测试工程师 | Cypress、Playwright、Selenium | 迭代周期 |
| 测试阶段 | 兼容测试 | 测试工程师 | BrowserStack、LambdaTest | 迭代周期 |
| 上线前 | 性能测试 | 测试工程师 | JMeter、K6、Locust | 2 天 |
| 上线前 | 安全测试 | 安全工程师 | OWASP ZAP、SonarQube | 1 天 |

**任务清单**：
- [ ] 制定详细的测试活动时间表
- [ ] 确定每个测试活动的开始和结束时间
- [ ] 确定每个测试活动的负责人
- [ ] 确定每个测试活动的依赖关系

**输出**：测试活动时间表

---

### 第七步：确定测试效果评价方法

**测试效果评价维度**：
1. **测试覆盖率**：代码覆盖率、功能覆盖率、需求覆盖率
2. **缺陷发现率**：每个测试阶段发现的缺陷数量
3. **缺陷修复率**：已修复缺陷 / 已发现缺陷
4. **缺陷密度**：缺陷数量 / 代码行数
5. **质量目标达成率**：质量目标的达成情况

**任务清单**：
- [ ] 定义测试覆盖率的目标（如：代码覆盖率 ≥ 80%）
- [ ] 定义缺陷发现率的阈值（如：系统测试阶段缺陷密度 ≤ 0.5/KLOC）
- [ ] 定义质量目标的量化标准
- [ ] 确定测试效果评价的频率（如：每个迭代结束后）

**输出**：测试效果评价标准

## 测试策略制定任务清单

**任务清单**：
- [ ] 第一步：分析质量目标
- [ ] 第二步：风险分析
- [ ] 第三步：确定测试范围和重点
- [ ] 第四步：选择测试类型
- [ ] 第五步：确定测试深度和广度
- [ ] 第六步：安排测试活动
- [ ] 第七步：确定测试效果评价方法
- [ ] 使用测试策略文档模板创建测试策略文档
- [ ] 将测试策略文档提交给团队评审
- [ ] 根据评审意见修改测试策略文档
- [ ] 测试策略文档必须获得批准后，才能进入下一步

**验收标准**：
- 测试策略文档已获得测试负责人和团队负责人的批准
- 测试策略文档包含完整的 8 个章节
- 测试策略文档中的所有决策都有明确的依据（质量目标、风险分析等）

**重要提醒**：
- **测试策略不是写测试用例**，测试策略是指导测试设计的方向和原则
- **测试策略应该基于质量目标和风险**，而不是基于个人经验或喜好
- **测试策略应该在技术规划之前制定**，因为测试策略会影响技术选型和架构设计
