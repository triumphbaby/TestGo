# TestExecution

本文档指导 AI 如何执行测试用例、记录测试结果、输出缺陷清单。

---

## 一、输入输出

### 输入

**第一轮测试**：
- 测试用例文档（`ModuleName_ID_Cases.md`）
- 测试数据文档（`ModuleName_ID_Data.md`）

**后续轮次测试（回归测试）**：
- 测试用例文档（`ModuleName_ID_Cases.md`）
- 测试数据文档（`ModuleName_ID_Data.md`）
- 上一轮测试问题清单（命名格式：`ModuleName_ID_Bugs_n.md`，n 为轮次编号）

### 输出

- 每轮测试执行记录（命名格式：`ModuleName_ID_Execution_n.md`，n 为轮次编号）
- 每轮缺陷清单（命名格式：`ModuleName_ID_Bugs_n.md`，n 为轮次编号）

### 验收标准

- 所有文档按照标准命名格式保存
- 每轮测试用例都已执行
- 每轮发现的缺陷都已记录
- 每轮缺陷修复都已验证

## 二、测试执行流程

### 2.1 收集上下文

使用 Read 工具**并行**读取以下文件：
- 测试用例文档（`ModuleName_ID_Cases.md`）
- 测试数据文档（`ModuleName_ID_Data.md`）
- 上一轮问题清单（回归测试时：`ModuleName_ID_Bugs_n.md`）

从测试用例文档中提取每个用例的：
- 用例编号、标题、优先级
- 测试活动阶段（系统测试/兼容测试/性能测试/安全测试）
- 测试场景、风险对应、模块
- 平台/环境（客户端/平台、网络环境、硬件环境）
- 前置条件（环境要求、开始路径、测试数据）
- 操作检验步骤、预期结果

从测试数据文档中获取：
- 测试数据编号、类型、数量
- 准备位置、准备方法
- 对应的测试用例

从上一轮问题清单中获取（回归测试时）：
- 已修复缺陷列表
- 待验证缺陷列表
- 已知限制

### 2.2 规划测试执行顺序

**按测试活动阶段排序**：
1. 系统测试
2. 兼容测试
3. 性能测试
4. 安全测试

**按优先级排序（同一阶段内）**：
1. P0 用例（核心用例，正向主要业务流程）
2. P1 用例（重要用例，核心功能和边界场景）
3. P2 用例（一般用例，辅助功能和异常场景）

**依赖关系处理**：
- 优先执行无依赖的基础用例（如登录、页面加载）
- 有依赖的用例在其前置用例通过后执行
- 如果前置用例失败，将依赖它的用例标记为 `skipped`

### 2.3 测试执行

#### 浏览器操控工具

通过 `chrome-devtools` MCP 工具操控浏览器（工具名以 `mcp__chrome-devtools__` 为前缀）：

- **导航**：`navigate_page` — 打开指定 URL
- **快照**：`take_snapshot` — 获取页面 accessibility snapshot，用于定位元素 ref
- **点击**：`click` — 通过 ref 点击元素
- **输入**：`fill` — 向输入框输入文本
- **填写表单**：`fill_form` — 批量填写多个表单字段（优先使用，减少调用次数）
- **截图**：`take_screenshot` — 截取当前页面截图
- **等待**：`wait_for` — 等待指定元素出现
- **视口**：`resize_page` — 设置浏览器视口大小
- **弹窗**：`handle_dialog` — 处理弹窗（确认/取消）

#### 前置检查

检查该用例的前置条件是否满足：
- 环境要求（服务状态、网络环境）
- 开始路径（页面入口、登录状态）
- 测试数据（数据准备是否就绪）

不满足则标记 `skipped` 并跳过。

#### 执行测试步骤

**核心原则：最小化工具调用次数。**

- **每个用例开始时**调用一次 `take_snapshot`，获取页面状态和元素 ref
- **复用 snapshot**：同一页面状态下的多个操作（点击、输入）共用一个 snapshot 的 ref，不要每次操作前都重新 snapshot
- **批量输入**：多个表单字段用 `fill_form` 一次性填写，不要逐字段 click + fill
- **仅在页面状态变化后**重新 snapshot：导航跳转、表单提交、弹窗触发等
- **连续操作**可以不加 snapshot，只在需要验证结果时 snapshot

#### 验证与截图

- 操作完成后调用 `take_snapshot` 检查预期结果（文本、元素、URL 等）
- 每个用例只在**最终验证点**截图一次，保存到 `screenshots/{testCaseId}.png`
- 失败用例额外截一张错误状态截图：`{testCaseId}-failed.png`

#### 异常处理

- **元素未找到**：调用 `wait_for` 等待最多 5 秒，再 snapshot，仍未找到则标记失败
- **页面超时 / JS 错误**：截图记录，标记失败
- **弹窗阻断**：用 `handle_dialog` 处理，记录弹窗内容

#### 结果判定

每个步骤记录：操作描述、预期结果、实际结果、通过/失败。
- 某步骤失败时继续执行后续步骤（除非页面不可恢复）
- 所有步骤通过 → `passed`
- 任一步骤失败 → `failed`
- 前置条件不满足 → `skipped`
- 不可恢复异常 → `error`

### 4.3 缺陷编号规范

**格式**：`BUG-XX-XXX`

**说明**：
- `BUG`：标识为缺陷
- `XX`：一级模块的单词缩写，用两个字母
- `XXX`：数字编号，从001开始递增

**示例**：
- `BUG-BK-001`：备份模块第1个缺陷
- `BUG-PT-005`：相册模块第5个缺陷

### 4.4 缺陷标题规范

**原则**：用简单的话语描述问题表现，能一眼看懂意思、了解问题全貌。

**规范要求**：
- 描述问题现象，而非推测原因
- 包含关键信息：模块、功能点、问题表现
- 避免模糊表述：如"有问题"、"不正常"、"报错"
- 简洁明了，不超过30字

**正确示例**：
- ✅ 备份页面点击开始按钮后无响应
- ✅ 相册列表加载时图片显示为空白
- ✅ 登录页面输入正确密码提示密码错误
- ✅ 文件上传过程中取消按钮无法点击

**错误示例**：
- ❌ 备份有问题（太模糊）
- ❌ 相册不正常（未描述具体问题）
- ❌ Bug（无任何信息）
- ❌ 系统报错xxx可能是因为参数传递不对（包含推测原因）

### 4.5 缺陷严重级别定义

| 级别 | 定义 | 处理时效 | 问题类型 |
|------|------|----------|----------|
| P0级 | 在需求和UI明确设计和说明的情况下与设计不符，及其他高危或阻塞问题 | 必须在当前版本修复且阻塞问题优先解决 | 实现偏差、安全问题、系统服务问题、主流程阻塞、核心服务响应慢、不明确需求 |
| P1级 | 非核心主流程功能、影响用户体验较小的问题 | 下个版本发布版本前必须解决 | 功能缺陷、未考虑场景问题 |
| P2级 | 非核心功能问题，影响部分用户体验或影响较小，问题场景较深 | 两个版本发布版本前必须解决 | 边界条件问题、轻微性能问题 |
| P3级 | 不影响功能使用的优化建议或需求建议 | 根据产品需求规划在后续版本解决 | 体验优化、文案优化、竞品参考、技术债务 |

#### P0级问题类型详解

**1. 实现偏差**：功能逻辑与需求文档描述不一致，UI还原度(按钮位置、文案、图片、图标)、交互与UI设计不一致、底层依赖库版本不一致
- 需求要默认设置10小时一次检测，实现是9小时一次
- 交互要点击热区进行跳转，实现是必须点击文字

**2. 安全问题**：可能导致系统被攻击或数据泄露的漏洞或数据丢失
- SQL注入漏洞，XSS跨站脚本攻击
- 未授权访问漏洞
- 敏感数据明文传输

**3. 系统服务问题**：服务不可用、系统无法继续使用
- 使用过程服务崩溃停止，系统卡死
- 机器异常重启关机断连
- 服务不可进入、服务提供的主流程受阻

**4. 主流程阻塞**：关键功能完全无法使用（需求提测可打回）
- 用户无法登录，OTA升级失败，核心业务流中断，穿透连接失败

**5. 核心服务响应慢**：响应超过3s（特殊场景需要给出理论依据）
- 核心页面相同条件下加载时长超过同类竞品，接口响应超时

**6. 不明确需求或争议**（小组长领头决定后修改其级别，开发不可直接拒绝）
- 分不清楚严重级别的
- 需求不明确的

#### P1级问题类型详解

**1. 功能缺陷**：非主流程功能存在阻塞或临界问题
- 文件上传特定层级或文件名路径从任务中心跳转会空白
- 搜索某些数据返回none等特定错误结果

**2. 未考虑场景问题**：产品、设计、开发、测试四方在设计阶段都未考虑到的场景问题，且不影响正常功能
- 特殊复杂场景下字体大小不一致
- 特殊复杂场景下鼠标悬停效果缺失

#### P2级问题类型详解

**1. 边界条件问题**：特定操作顺序或数据下出现的异常
- 连续快速点击5次及以上导致展示问题

**2. 轻微性能问题**：可感知但不阻塞操作的性能缺陷
- 列表滚动轻微卡顿
- 视频最大化最小化或窗口移动过程掉帧

#### P3级问题类型详解

**1. 体验优化**：提升用户体验的改进建议
- 增加操作快捷键，优化表单填写流程

**2. 文案优化**：提示信息不够友好或准确
- 错误提示语不够明确，帮助文案描述模糊

**3. 竞品参考**：借鉴行业最佳实践的优化建议
- 参照竞品优化搜索筛选条件
- 改进数据可视化方式

**4. 技术债务**：代码质量或架构优化建议
- 重构历史遗留代码
- 优化数据库查询效率
- 优化log打印规范

### 4.6 缺陷类型定义

| 类型 | 说明 |
|------|------|
| 功能缺陷 | 功能未按预期工作 |
| 界面缺陷 | UI 展示异常、样式错误 |
| 性能问题 | 响应慢、资源占用高 |
| 兼容性问题 | 特定环境下的异常 |
| 安全问题 | 权限漏洞、数据泄露风险 |

以下是一个完整的缺陷记录示例：

## 五、回归测试指引

### 5.1 回归测试触发条件

- 上一轮缺陷修复完成后
- 需求变更导致用例调整后
- 环境升级或配置变更后

### 5.2 回归测试范围

**必选**：
- 上一轮发现的缺陷对应用例（验证修复）
- 上一轮标记为 failed 的用例

**按需选择**：
- 与修复缺陷相关的模块用例
- P0 核心用例（确保核心功能正常）
- 本次需求变更影响范围的用例

### 5.3 回归测试执行

1. 读取上一轮问题清单（`ModuleName_ID_Bugs_{n-1}.md`）
2. 确认已修复缺陷列表
3. 按执行顺序重新执行相关用例
4. 记录验证结果：
   - 验证通过：缺陷状态更新为"已修复"
   - 验证失败：缺陷状态更新为"修复失败"，需重新修复

## 六、注意事项

- **严格按执行顺序**：测试活动阶段 → 优先级
- **前置条件必须满足**：不满足则跳过并记录原因
- **最小化工具调用**：复用 snapshot，批量操作
- **截图规范**：每个用例最终验证点截图一次，失败用例额外截图
- **结果记录完整**：每个步骤都要记录预期/实际结果
- **缺陷编号唯一**：同一模块内编号不重复
- **文档命名规范**：严格按照格式命名
- **多轮测试追溯**：回归测试关联上一轮问题清单
