# 测试用例共享规范

本文档定义测试用例的通用字段规范、输出格式、优先级定义和颗粒度规则，供所有测试用例 Skill 引用。

## 一、输出规范

### 技术方案
- 使用 Python + openpyxl 库生成 xlsx 文件
- 文件命名：`{需求文档名}_测试用例.xlsx`，保存在需求文档同目录下
- 如需求文档为粘贴文本，命名为 `测试用例_{YYYYMMDD_HHmmss}.xlsx`，保存在当前工作目录

### 表格结构
- 按一级模块分 Sheet（仅一个模块时 Sheet 名为"测试用例"）
- 表头列顺序：

| 用例编号 | 所属模块 | 用例名称 | 优先级 | 测试活动阶段 | 风险对应 | 前置条件 | 步骤描述 | 预期结果 |
|---------|---------|---------|-------|-----------|---------|---------|---------|---------|

### 样式要求
- 表头行：加粗、浅灰背景色（RGB: 217,217,217）
- 列宽：自适应内容，最小 15 字符，最大 60 字符
- 单元格：文本自动换行，垂直方向顶部对齐
- 「用例编号」列（A列）默认隐藏，需要时手动取消隐藏

## 二、用例字段定义

### 用例编号
- 功能/兼容/专项/升级用例：`TC_{模块缩写}_{三位序号}`，如 `TC_LOGIN_001`
- Bugfix验证用例：`TC_BF_{模块缩写}_{三位序号}`，如 `TC_BF_SEARCH_001`（BF前缀区分验证类型）

### 用例名称
- 使用"验证..."句式，简明描述测试目的

### 优先级
| 等级 | 定义 | 颗粒度 |
|-----|------|--------|
| P0 | 阻塞级：核心业务端到端正向流程 | 粗颗粒度，覆盖多步骤主流程 |
| P1 | 高优先级：单功能点正向/反向验证 | 中等颗粒度，一个用例一个功能点 |
| P2 | 中优先级：异常处理、性能、安全等专项场景 | 细颗粒度，针对特定条件精确验证 |

### 测试活动阶段
每条用例必须标注所属的测试活动阶段（与 Skill 类型对应）：

| 阶段 | 说明 | 对应 Skill |
|------|------|-----------|
| 系统测试 | 正常/异常/边界/网络异常/服务异常/并发场景 | 功能测试用例 |
| 兼容测试 | 不同平台、浏览器、分辨率、网络环境 | 兼容性测试用例 |
| 性能测试 | 响应速度、大数据量、频繁操作、资源泄漏 | 专项测试用例 |
| 安全测试 | XSS/SQL注入/鉴权绕过/数据泄露/CSRF | 专项测试用例 |
| 升级测试 | DB迁移、OTA升级、刷机初始化 | 升级刷机测试用例 |

### 风险对应
- 关联 TestStrategy 文档中识别的风险项，增强用例可追溯性
- 格式：`R{编号}（风险简述）`，如 `R1（用户认证安全性问题）`
- 无对应风险项时填"无"
- 一条用例可关联多个风险项，用逗号分隔

### 所属模块
- 完整格式：`版本用例/{版本}/kaixuan/{需求}/{用例类型}/{模块路径}`
  - **版本**：当前测试版本号（如 2.4.0），不带 V 前缀，每次生成前向用户确认
  - **kaixuan**：固定人员，无需确认
  - **需求**：需求名称或编号，每次生成前向用户确认
  - **用例类型**：功能测试 / 兼容性测试 / 专项测试 / 升级测试（与生成的xlsx文件类型对应）
  - **模块路径**：`一级模块 > 二级模块`（仅一级则省略层级符号）
- 示例：`版本用例/2.4.0/kaixuan/CopySource拷贝学习继承/功能测试/文件管理 > 拷贝AI学习继承`

### 前置条件
必须包含以下三部分（无则写"无"）：
- **环境要求**：测试环境状态（上下游服务依赖、设备状态、网络条件等）
- **开始路径**：用户界面入口路径或 URL
- **测试数据**：所需数据及准备方法（数据库记录、文件、接口预置等）

Bugfix验证用例额外必须包含：
- **Bug环境**：Bug的原始触发环境（浏览器版本、分辨率、设备等）
- **修复版本**：修复所在的代码版本/分支/commit

### 步骤描述
- 按编号顺序书写（1. 2. 3.）
- 使用祈使句（"点击【xx】按钮"、"在'名称'输入框中输入'xxx'"）
- 操作动作与检验动作分开编号，检验步骤标记为"检验步骤N：检查xxx"

### 预期结果
- 与检验步骤严格一一对应，格式："检验步骤N：期望结果描述"
- 结果必须明确、可量化、可验证
- 每条预期结果至少包含以下一种检查：
  - **界面/接口校验**：UI 变化、提示信息、接口返回数据
  - **DevTools校验**：CSS计算属性值、DOM结构、Network请求/响应（Bugfix验证常用）
  - **日志校验**：明确日志路径 + 期望关键字（或不应出现的错误）
  - **数据库校验**：明确表名 + 字段 + 期望状态
  - **Figma基线校验**（仅当存在UI Spec时）：引用 UI Spec 中的具体视觉规格值（颜色HEX、字体、尺寸）或参考图文件名

## 三、颗粒度判断规则

满足以下任一条件时应拆分用例：
1. 用例失败时无法立刻定位问题功能点
2. 执行步骤超过 15 步
3. 不同验证点需要完全不同的前置条件
4. 用例描述中包含"并且"、"同时"等多验证点信号词

## 四、自检清单

生成用例后，逐项检查：
1. 是否覆盖了需求文档中所有功能点
2. 是否包含正向和反向用例
3. 是否存在冗余用例（可合并的相似场景）
4. 预期结果是否都包含至少一种检查类型（界面/日志/数据库）
5. 前置条件是否足够详细，其他测试人员可独立执行
6. 用例编号是否连续且无重复
7. 测试活动阶段是否已标注
8. 风险对应是否已填写（无则明确标注"无"）
9. 如有 UI Spec，视觉相关用例的预期结果是否引用了具体的颜色/字体/尺寸值（而非模糊描述如"颜色正确"）

## 六、用例库管理

### 版本管理
- 每次用例更新后保留历史版本
- 文件命名规范：`{需求文档名}_{用例类型}_v{版本号}.xlsx`
  - 示例：`AI搜索优化_功能测试用例_v1.0.xlsx`、`AI搜索优化_功能测试用例_v1.1.xlsx`
- 主版本号：用例结构或大范围变更时递增
- 次版本号：小范围修改或新增时递增

### 用例状态标签
当基于已有用例库进行迭代更新时，在用例名称前添加状态标签：

| 标签 | 含义 | 使用场景 |
|------|------|---------|
| `[NEW]` | 本次新增的用例 | 新增功能点对应的新用例 |
| `[MODIFIED]` | 本次修改的用例 | 因需求变更而调整的已有用例 |
| `[UNCHANGED]` | 未变更的用例 | 仅在增量输出中使用，表示保持不变 |
| `[DEPRECATED]` | 已失效的用例 | 因功能下线或需求取消而失效的用例 |

### 影响范围追踪
对于优化需求（非全新功能），用例文档开头应记录：
- **本次需求影响的模块列表**
- **回归测试范围及理由**
- **与上一版本用例库的差异摘要**（新增 X 条、修改 X 条、废弃 X 条）
